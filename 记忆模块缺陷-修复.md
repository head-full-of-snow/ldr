# MemoryError in History/Memory Features - Analysis and Fix Plan

## Context

User encountered a `MemoryError` when using history/memory-related features. The error occurred at `cursor.execute()` in SQLAlchemy:

```
MemoryError at sqlalchemy/engine/base.py in _exec_single_context
```

**Root cause**: Multiple database queries load excessive ORM objects into memory at once.

## Root Cause Analysis

### Issue 1: `history_routes.py:54-65` — Loading entire `ResearchHistory` ORM object

```python
db_session.query(
    ResearchHistory,  # Loads all columns, including report_content (Text large field)
    func.count(Document.id).label("document_count"),
)
```

Although limited to 200 rows (max 500), **`ResearchHistory` is a complete ORM object** including `report_content` (Text field). Each detailed research report can be large (several MB). 200 rows x several MB can exceed memory limits.

### Issue 2: `library_service.py:561-569` — No limit on full history load

```python
session.query(ResearchHistory.id, ResearchHistory.title, ResearchHistory.query)
    .order_by(ResearchHistory.created_at.desc())
    .all()  # No limit!
```

Dropdown selector loads **all** history records without pagination.

### Issue 3: `storage/database.py:131-134` — `list_reports()` without pagination

```python
results = self.session.query(ResearchHistory).filter(
    ResearchHistory.report_content.isnot(None)
).all()  # Loads all researches with reports, including large fields
```

Although only a few fields are used in the return value, the query loads complete `ResearchHistory` objects.

### Issue 4: `library_service.py:360-391` and `library_rag_service.py:1747-1750` — Collection queries without limit

- Load documents by collection: `.all()` without limit
- Load all chunks for a collection: `.all()` without limit

## Fix Plan

### Fix 1: `src/local_deep_research/web/routes/history_routes.py`

**Problem**: `query(ResearchHistory, ...)` loads `report_content` and other large fields, but the response only uses `id, title, query, mode, status, created_at, completed_at, duration_seconds, research_meta, chat_session_id`.

**Solution**: Query only the needed columns (exclude large fields):

```python
results = (
    db_session.query(
        ResearchHistory.id,
        ResearchHistory.title,
        ResearchHistory.query,
        ResearchHistory.mode,
        ResearchHistory.status,
        ResearchHistory.created_at,
        ResearchHistory.completed_at,
        ResearchHistory.duration_seconds,
        ResearchHistory.research_meta,
        ResearchHistory.chat_session_id,
        func.count(Document.id).label("document_count"),
    )
    .outerjoin(Document, Document.research_id == ResearchHistory.id)
    .group_by(ResearchHistory.id)
    .order_by(ResearchHistory.created_at.desc())
    .limit(limit)
    .offset(offset)
    .all()
)
```

Then change result processing from ORM attribute access to tuple unpacking.

### Fix 2: `src/local_deep_research/research_library/services/library_service.py`

**Problem**: `get_research_list_for_dropdown()` has no limit.

**Solution**: Add `.limit(100)` (dropdown does not need all data).

### Fix 3: `src/local_deep_research/storage/database.py`

**Problem**: `list_reports()` loads complete `ResearchHistory` objects.

**Solution**: Query only needed columns (same approach as Fix 1).

### Fix 4: `src/local_deep_research/research_library/services/library_service.py:360-391`

**Solution**: Add a reasonable limit (e.g., 500).

### Fix 5: `src/local_deep_research/research_library/services/library_rag_service.py`

**Solution**: Add limit to chunk queries or use `.yield_per()` for streaming.
