一、项目整体架构

Local Deep Research (LDR) 是一个 AI 驱动的深度研究助手，核心能力是将一个复杂问题自动拆解为多轮搜索、分析、综合，最终生成带完整引用的研究报告。系统采用 策略模式 (Strategy Pattern) 架构，内置 30+ 研究策略，支持 28 种搜索引擎、10 种 LLM 提供商。

用户浏览器 (localhost:5000) │ ▼ Flask 后端 (REST API + WebSocket) ├── 认证模块 (Flask-Login, CSRF, 限流) ├── 研究队列 (QueueProcessorV2, 多用户串行化锁) ├── 策略选择器 (StrategyFactory → 30+ 策略) │ └── 迭代循环: 生成子问题 → 并行搜索 → 分析结果 → 综合报告 ├── 引用处理器 (CitationHandler) ├── 跨引擎过滤器 (CrossEngineFilter) │ ├──► LLM 提供商 (Ollama / OpenAI / Anthropic / LM Studio / LlamaCpp / ...) ├──► 搜索引擎 (arXiv / PubMed / Wikipedia / SearXNG / Brave / DuckDuckGo / ...) │ └── 存储 ├── 每用户加密数据库 (SQLCipher AES-256) ├── FAISS 向量索引 (知识库) └── 文件存储 (研究输出 / 文档库)

研究流程

  1. 输入阶段：用户提交研究问题
  2. 迭代研究：LLM 生成子问题 → 多引擎并行搜索 → 结果分析与置信度评估，循环直至信息充足
  3. 综合阶段：跨引擎结果去重/排序/LLM 相关性评分 → 引用标注 → 生成结构化报告
  4. 输出阶段：Markdown / PDF 报告，支持导出到知识库

二、前后端结构

是的，项目有清晰的前后端分离架构。

后端

基于 Flask + Flask-SocketIO，提供 REST API 和 WebSocket 实时推送。核心模块：

模块路径职责
App Factoryweb/app_factory.py应用初始化、中间件注册、蓝图挂载
研究队列web/queue/processor_v2.py多用户研究任务调度，每用户串行化锁
认证系统web/auth/用户注册/登录、会话管理、CSRF 防护
API 路由web/routes/研究、设置、历史、指标等 REST 端点
搜索引擎web_search_engines/28 种搜索引擎适配器 + 自适应限流
LLM 注册表llm/llm_registry.py线程安全的 LLM 提供商管理
加密数据库database/encrypted_db.py每用户 SQLCipher 数据库管理

前端

基于 Vite + 原生 JavaScript 构建（非 SPA 框架），通过 Flask 模板渲染 + WebSocket 接收实时研究进度。构建产物为 static/dist/ 下的 JS/CSS 文件。

三、核心创新点

3.1 自适应限流器 — 基于强化学习的速率控制

AdaptiveRateLimitTracker（位于 web_search_engines/rate_limiting/tracker.py，31KB）是项目中最大的创新之一：

  • 探索-利用机制 (ε-greedy)：10% 概率尝试更短的等待时间（探索），90% 使用历史学习到的最优值（利用），判断搜索引擎 API 限制是否已放宽
  • 指数移动平均：learning_rate=0.3，使用成功请求的中位等待时间持续更新估计值
  • 三种配置画像：conservative（探索率 5%、学习率 20%）、balanced（默认）、aggressive（探索率 20%、学习率 50%）
  • 持久化学习：将学到的等待时间存入每用户加密数据库，下次启动自动加载，并按 decay_per_day=0.95 衰减旧数据
  • 乐观初始化：未知引擎从 0.1 秒等待开始，避免过度保守

3.2 30+ 可插拔研究策略

通过策略模式将研究逻辑抽象为 BaseSearchStrategy 基类，实现 30+ 种策略，覆盖不同研究场景：

策略核心思路
FocusedIterationStrategy聚焦迭代，SimpleQA 基准测试 ~95% 准确率
LangGraphAgentStrategyLLM 自主决定搜索什么、用哪个引擎、何时综合
SourceBasedSearchStrategy基于来源的迭代搜索，默认策略
DualConfidenceWithRejectionStrategy双置信度 + 早期拒绝：负面证据超 25% 立即排除
IterativeReasoningStrategy维护 KnowledgeState 数据结构，逐步推理
AdaptiveDecompositionStrategy自适应问题分解
RecursiveDecompositionStrategy递归子问题分解
ParallelSearchStrategy最大化并行搜索吞吐量
IterDRAGStrategy迭代检索增强生成

3.3 两阶段搜索检索

BaseSearchEngine（search_engine_base.py）实现两阶段检索：

  1. 预览阶段：批量获取大量结果的摘要信息（标题、片段、URL）
  2. 全文阶段：仅对 LLM 判定为相关的结果获取完整内容

LLM 相关性过滤采用批量并行处理（relevance_filter_batch_size=5，最大 10 个并行批次），大幅节省带宽和 Token 消耗。

3.4 跨引擎结果融合

CrossEngineFilter（cross_engine_filter.py）将 28 种搜索引擎的结果进行统一处理：

  • LLM 驱动的相关性评分与重排序
  • 跨引擎去重（URL 级别 + 内容级别）
  • 可配置最大结果数（默认 100）和 LLM 上下文窗口（最多展示 30 条给 LLM）
  • 结果少于 10 条时跳过 LLM 过滤，避免不必要的开销

3.5 知识闭环系统

研究 → 下载来源 → 知识库 → 分块/嵌入 → FAISS 向量索引 → 作为搜索引擎复用

每次研究的有价值来源（arXiv 论文、PubMed 文章、网页）可一键下载到加密知识库。LDR 自动提取文本、分块、生成嵌入向量、构建 FAISS 索引。下次研究时可以将自己的文档集合作为搜索引擎，实现知识复利积累。

3.6 自适应候选探索器

AdaptiveExplorer（candidate_exploration/adaptive_explorer.py）会学习哪种搜索策略对当前查询类型最有效：

  • 四种初始策略：直接搜索、同义扩展、类别探索、相关术语
  • 跟踪每种策略的 attempts、candidates_found、quality_sum
  • 阈值 5 次搜索后开始自适应，选择每尝试获得最多候选的策略

四、性能优化方法

4.1 并行搜索 (ThreadPoolExecutor)

每轮迭代生成的所有子问题通过 ThreadPoolExecutor 并行发送到多个搜索引擎。SourceBasedSearchStrategy 中，单轮的所有问题搜索同步并发执行，而非串行等待。

4.2 连接池管理

每用户独立的 SQLAlchemy QueuePool：

  • pool_size=20，max_overflow=40，pool_timeout=10s
  • 每用户独立数据库引擎，互不干扰
  • SQLCipher 连接使用 WAL 模式优化并发读写

4.3 资源清理的纵深防御

多层机制确保无资源泄漏：

  • @thread_cleanup 装饰器 — 线程退出时自动清理
  • finally 块 — 研究流程异常时兜底释放
  • teardown_appcontext — Flask 请求结束时清理
  • 定时连接池 dispose（每 30 分钟）— 清理长时间闲置连接
  • 登出级联 — 用户登出时立即释放所有资源
  • 过期会话清理 — 定期扫描并清理僵尸会话

4.4 研究队列处理器

QueueProcessorV2 管理多用户研究任务的并发：

  • 每用户串行化锁：_user_critical_locks 确保同一用户不会同时执行多个研究任务
  • 指数退避重试：INITIAL_RESEARCH_LOOKUP_DELAY=0.5s，RETRY_BACKOFF_MULTIPLIER=2，最多重试 3 次
  • 启动重试限制：SPAWN_RETRY_LIMIT=3，超过后放弃排队中的任务

4.5 WebSocket 实时进度流

通过 Flask-SocketIO 将研究进度实时推送到浏览器（问题生成、搜索引擎调用、结果分析、报告综合），用户无需轮询，减少无效请求。

4.6 LLM 批量并行过滤

搜索结果的相关性评分不是逐条调用 LLM，而是批量处理（默认 5 条一批，最大 10 批并行），将 LLM 调用次数从 N 降低到 N/5。

4.7 文件描述符预算管理

SQLCipher WAL 模式每个连接占用 2 个 FD。系统精确计算稳态（用户数 × 41 FDs）和峰值（用户数 × 121 FDs），确保不超出系统限制。

五、安全设计

5.1 每用户加密数据库

每个用户拥有独立的 SQLCipher 数据库，使用 AES-256 加密。密钥由用户登录密码通过 PBKDF2 派生，配合独立 salt 文件。即使服务器管理员也无法读取用户数据（零知识架构）。

5.2 安全头中间件

security_headers.py 注入 CSP、X-Content-Type-Options、X-Frame-Options 等安全响应头。CSP 配置针对 Socket.IO 兼容性保留了 unsafe-inline，但移除了 unsafe-eval。

5.3 多层限流

接口限流策略
全局默认5000 次/小时，50000 次/天
登录5 次/15 分钟
注册3 次/小时
设置修改30 次/分钟
搜索引擎自适应限流（见 3.1 节）

5.4 供应链安全

Docker 镜像使用 Cosign 签名，包含 SLSA 来源证明和 SBOM。代码通过 CodeQL、Semgrep、Bearer、DevSkim、Gitleaks 等 10+ 种安全扫描工具持续审计。
